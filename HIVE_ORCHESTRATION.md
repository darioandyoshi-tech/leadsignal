3. **Stage 2.5 – Integrity Gate** — Verify all claims have locator anchors (see `VERIFICATION_PROTOCOL.md`).
4. **Stage 3 – Plan** — Detailed architecture + Devil's Advocate round.
5. **Stage 4 – Execute** — Implementation.
6. **Stage 4.5 – Integrity Gate** — Full claim verification + cross-model check.
7. **Stage 5 – Review** — Multi-node audit (Security, Performance, Business Alignment).
8. **Stage 6 – Revise** — Incorporate feedback with concession protocol.
9. **Stage 7 – Finalize** — Style calibration check + final Gate 4.5.
10. **Stage 8 – Compound** — Extract reusable skill, update `BUSINESS_KNOWLEDGE.md`, LEV capture.

**Mandatory Checkpoints:** User confirmation required at Stage 1, 2.5, 4.5, and 8.

## Team Architecture Patterns (from revfactory/harness)

The HIVE uses these 6 architectural patterns when designing agent teams:

| Pattern | Description | HIVE Usage |
|---------|-------------|------------|
| **Pipeline** | Sequential dependent tasks | Used for linear workflows (e.g., Research → Plan → Execute → Review) |
| **Fan-out/Fan-in** | Parallel independent tasks | Used when multiple Nodes work in parallel (e.g., Finance + AWM Dev + PulseWatch Growth) |
| **Expert Pool** | Context-dependent selective invocation | Used when specialized expertise is needed on demand |
| **Producer-Reviewer** | Generation followed by quality review | Core pattern — aligns with Verification Protocol + Devil’s Advocate |
| **Supervisor** | Central agent with dynamic task distribution | Yoshi (Nexus) role |
| **Hierarchical Delegation** | Top-down recursive delegation | Used when breaking down complex missions into sub-agents |
